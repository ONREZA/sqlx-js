# Upgrade to 0.20.0

Status: unreleased.

This guide applies to applications upgrading from `0.19.x` to `0.20.0`.
Because sqlx-js is still pre-1.0, replacing the runtime driver is released as a
minor version under the repository's Rust-style pre-1.0 SemVer policy.

## Summary

sqlx-js no longer depends on Postgres.js. Prepare, migrations, raw clients, and
managed runtime clients now share the integrated PostgreSQL wire
implementation.

The new runtime:

- is ESM-only and supports Node 24, Bun 1.3, and Deno 2.9 or newer;
- requires PostgreSQL 16 or newer;
- returns PostgreSQL `int8` as native `bigint`;
- uses a lazy bounded pool with FIFO admission, idle eviction, and maximum
  connection lifetime;
- serializes every connection, including concurrent calls made inside one
  transaction;
- sends parameterized queries through the unnamed extended protocol;
- cancels through PostgreSQL `CancelRequest`;
- discards broken connections and reconnects later operations without replaying
  statements whose outcome may be unknown.

See the complete
[Postgres.js compatibility matrix](../postgres-js-feature-matrix.md) for
supported, missing, and permanently rejected surfaces.

## Replace Postgres.js options

`createClient(...)` and `createSqlClient(...)` no longer accept arbitrary
Postgres.js options. Replace the supported connection and pool settings with
the typed sqlx-js names:

| Postgres.js option | sqlx-js option |
| --- | --- |
| `max` | `max` |
| `connect_timeout` in seconds | `connectTimeoutMs` in milliseconds |
| `idle_timeout` in seconds | `idleTimeoutMs` in milliseconds |
| `max_lifetime` in seconds | `maxLifetimeMs` in milliseconds |
| `connection.application_name` | `applicationName` |
| `connection.options` | `startupOptions` |
| `connection.role` | `role` |
| password function | `password` |
| `onnotice` | `onNotice` |
| numeric `types` | numeric `types` |

For example:

```ts
const db = createSqlClient(process.env.DATABASE_URL, {
  max: 20,
  connectTimeoutMs: 10_000,
  idleTimeoutMs: 60_000,
  maxLifetimeMs: 30 * 60_000,
  applicationName: "api",
  statementTimeoutMs: 5_000,
});
```

Move host, port, username, password, database, and TLS configuration into
`DATABASE_URL`. The runtime recognizes `sslmode`, certificate paths,
`application_name`, `options`, `role`, `connect_timeout`, and
`statement_timeout`.

The following Postgres.js options have no direct replacement:

- `prepare` and `max_pipeline`: sqlx-js always uses unnamed statements and does
  not pipeline automatically;
- `transform`: perform application/domain mapping explicitly;
- `fetch_types`: required runtime type discovery is owned by sqlx-js;
- `debug` and `onparameter`: use sqlx-js query/lifecycle observers for the
  supported telemetry contract;
- `socket`, multiple hosts, and `target_session_attrs`: use a stable database
  endpoint or proxy;

## Review raw client use

`createClient(...)` now exposes the intentionally small sqlx-js raw contract:

- `unsafe(query, params?)`;
- pending-query `execute()`, `cancel()`, and `values()`;
- `begin(options?, callback)`;
- `array(...)`, `json(...)`, and `typed(...)`;
- `end()`.

The optional second argument to raw `array(value, arrayOid)` is the PostgreSQL
array OID, not its element OID. Omit it when the query's `Describe` result can
provide the parameter type. `typed(value, oid)` continues to take the exact
value OID.

Replace Postgres.js `end({ timeout })` calls with `end()`. Raw shutdown always
interrupts its owned startup, queued work, and active sockets immediately;
managed graceful/forced deadlines belong to `createSqlClient().close(...)`.

Tagged templates, dynamic query builders, cursor/COPY helpers, savepoint
callbacks, `LISTEN`/`NOTIFY`, logical replication subscriptions, connection
reservation, and Postgres.js result internals are not reproduced. Migrate
application code to the typed `createSqlClient(...)` surface or keep the
specialized workload on a separate purpose-built client. Sending a COPY
streaming statement through `unsafe(...)` fails explicitly and discards that
connection instead of waiting for an unsupported stream.

## Runtime behavior changes

PostgreSQL `int8` values now decode to native `bigint` by default. Remove
application-side conversions that expected a decimal string.

PostgreSQL `date`, `timestamp`, and `timestamptz` contracts are now
`PgTemporal`, defined as `Date | "infinity" | "-infinity"`. Infinite values
round-trip as literals instead of becoming an invalid `Date`. Other PostgreSQL
temporal values outside JavaScript's finite `Date` range now fail decoding
explicitly.

Anonymous PostgreSQL `record` results now generate `string`, matching the
record literal returned by the text protocol. PostgreSQL `void` results decode
to `undefined` instead of an empty string.

`count` and `command` remain available on raw result arrays but are
non-enumerable, so row-array equality and serialization are not polluted by
driver metadata.

Concurrent operations can use different pool connections and have no global
ordering guarantee. Operations sharing one transaction connection execute
strictly in call order. Automatic per-connection pipelining is intentionally
unsupported.

No query is retried after dispatch. A connection failure or managed deadline
can leave the server-side outcome unknown; callers must use transaction and
idempotency semantics appropriate to the operation.

## Rollout and rollback

Roll out this change as an application release. The query-cache JSON format
does not change, but the generator revision increases because built-in output
contracts changed for temporal and anonymous record OIDs. Run a live
`sqlx-js prepare` with 0.20.0 and commit the updated manifest, cache entries,
and declarations. The revision also covers corrected and expanded stable
built-in OID mappings. An old cache cannot be upgraded with
`prepare --offline`.

Before deployment, inventory direct `createClient(...)` usage and removed
Postgres.js options. Exercise cancellation and reconnect against the same
database endpoint or pooler mode used in production.

Rollback requires restoring the previous sqlx-js version and its Postgres.js
option names together. Do not deploy configuration understood only by one
runtime version across mixed application replicas.

## Verification checklist

1. Run `bunx tsc --noEmit`.
2. Run `bun test tests --timeout 120000`.
3. Run `bun run test:runtime-boundary`.
4. Run `bun run test:node-package` with `DATABASE_URL`.
5. Run `bun run test:deno-package` with `DATABASE_URL` when Deno is deployed.
6. Run the application typecheck and its database integration tests.
