# Upgrade to 0.17.0

Status: unreleased.

This guide applies to applications upgrading from `0.16.x` to `0.17.0`.
Because sqlx-js is still pre-1.0, this CLI cleanup is released as a minor
version under the repository's Rust-style pre-1.0 SemVer policy.

## Summary

The CLI now separates the daily provider-aware workflow from provider-specific
deployment and migration operations:

- `dev` and `verify` select built-in migrations or pgschema from
  `sqlx-js.config.*`;
- `migrate` owns only built-in migration files and target history;
- `pgschema` owns the managed pgschema binary and target plan/apply operations;
- `snapshot` owns generated schema snapshots and manifests;
- `prepare` remains the query-artifact engine and no longer applies migrations.

## Command migration

| 0.16.x command | 0.17.0 command | Notes |
| --- | --- | --- |
| `sqlx-js migrate dev` | `sqlx-js dev` | Provider-aware shadow build; writes generated query artifacts |
| `sqlx-js db dev` | `sqlx-js dev` | Uses `schema.provider = "pgschema"` |
| `sqlx-js migrate verify` | `sqlx-js verify` | Provider-aware shadow verification; does not write |
| `sqlx-js db verify` | `sqlx-js verify` | Uses `schema.provider = "pgschema"` |
| `sqlx-js db install` | `sqlx-js pgschema install` | Installs the pinned pgschema binary |
| `sqlx-js db check` | `sqlx-js doctor` | Full-project diagnostic, not a binary-only probe; it also checks generated artifacts and the target database |
| `sqlx-js db plan` | `sqlx-js pgschema plan` | Explicit target-database deployment plan |
| `sqlx-js db apply` | `sqlx-js pgschema apply` | Explicit target-database apply |
| `sqlx-js schema dump` | `sqlx-js snapshot dump` | Writes the schema snapshot and optional manifest |
| `sqlx-js schema check` | `sqlx-js snapshot check` | Compares the committed snapshot with a live database |

The removed command names are not retained as aliases. Update package scripts,
CI workflows, shell scripts, editor tasks, and operator runbooks atomically
with the package upgrade.

## Provider-aware development and verification

For built-in migrations:

```bash
sqlx-js migrate add add_users
sqlx-js dev --strict-inference
sqlx-js verify --strict-inference
sqlx-js migrate run --dry-run
sqlx-js migrate run
```

For pgschema:

```bash
sqlx-js pgschema install
sqlx-js dev --strict-inference
sqlx-js verify --strict-inference
sqlx-js pgschema plan -- --output-json plan.json
sqlx-js pgschema apply -- --plan plan.json --auto-approve
```

Run `sqlx-js doctor` after generated artifacts and the target environment are
ready. It is a full-project diagnostic, not an installation-only pgschema
probe.

`dev` and `verify` never apply DDL to `DATABASE_URL`. They create an automatic
disposable shadow database, use `--shadow-admin-url`, or clear an explicitly
provided disposable `--shadow-url`.

`prepare --shadow-url` and its accompanying `--migrations` option have been
removed. Use provider-aware `dev` to build a schema and write artifacts,
`verify` to compare them, or set `DATABASE_URL` explicitly when the advanced
live `prepare` / `prepare --verify` path is required.

## CI behavior changed

`sqlx-js ci` is now a source-of-truth PR gate for both providers:

1. run provider-aware `verify --strict-inference`;
2. run `prepare --check --strict-inference`.

The 0.16.x pgschema CI path also required the target database to have an empty
pgschema plan and verified query artifacts against that target. That prevented
the aggregate command from validating a proposed `schema.sql` before
deployment. Target drift is now intentionally explicit:

```bash
# pgschema deployment gate
sqlx-js pgschema plan -- --output-json plan.json

# built-in migration deployment gate
sqlx-js migrate run --dry-run --json
```

`ci` no longer runs `snapshot check` automatically. Projects that commit a
snapshot should keep the target-specific check as a separate deployment step:

```bash
sqlx-js snapshot check
```

`snapshot dump/check --shadow-url` is now read-only. It no longer applies
built-in migrations implicitly, which was incorrect for pgschema projects.
Build the configured schema first with `dev --shadow-url`, then point
`snapshot dump/check --shadow-url` at that same disposable database.
Without the explicit flag, snapshot commands always inspect `DATABASE_URL`;
they no longer switch implicitly when `SHADOW_DATABASE_URL` is present.

## Generated package scripts

Newly scaffolded projects receive:

```json
{
  "scripts": {
    "sqlx:dev": "sqlx-js dev --strict-inference",
    "sqlx:verify": "sqlx-js verify --strict-inference",
    "sqlx:check": "sqlx-js prepare --check",
    "sqlx:ci": "sqlx-js ci"
  }
}
```

`init` never overwrites existing scripts. Existing projects must update
`sqlx:verify` manually if it still points to
`sqlx-js prepare --verify --strict-inference`, and add `sqlx:dev` if desired.
`prepare --verify` remains available for the narrower advanced case where
fresh query artifacts must be compared against a specifically supplied live
database rather than the configured schema source.

## Generated artifacts

This release does not change the cache or declaration format. Regeneration is
not required solely because command names changed. The normal `verify` gate
should still run against the configured schema source before deployment.

## Rollout checklist

1. Replace every removed command using the migration table above.
2. Update `sqlx:dev` and `sqlx:verify` package scripts.
3. Separate source verification (`sqlx-js ci`) from target deployment drift.
4. Keep `sqlx-js snapshot check` as an explicit step when snapshots are
   committed.
5. Run `sqlx-js doctor`, `sqlx-js verify --strict-inference`,
   `sqlx-js ci`, the project typecheck, and the project test suite.
6. For pgschema deployments, review a fresh `pgschema plan` before apply.

Do not deploy automation that uses the 0.17.0 command names while its installed
sqlx-js version can still resolve to `0.16.x`.
