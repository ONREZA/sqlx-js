# Upgrade to 0.18.0

Status: unreleased.

This guide applies to applications upgrading from `0.17.x` to `0.18.0`.
Because sqlx-js is still pre-1.0, the function-contract feature is released as
a minor version under the repository's Rust-style pre-1.0 SemVer policy.

## Summary

`KnownFunctions` and schema snapshots now preserve PostgreSQL routine security
and planner metadata:

- volatility and parallel safety;
- `SECURITY DEFINER` and `LEAKPROOF`;
- owner identity and whether the owner is a superuser;
- effective `EXECUTE TO PUBLIC`;
- function-local `search_path`;
- extension ownership.

`prepare` emits non-blocking `function-contract` warnings for high-risk
application-owned declarations. The same warnings are reproduced from the
committed cache by `prepare --check` and `prepare --offline`.

## Regenerate committed artifacts

The function-cache, declaration-generator, and schema-snapshot formats changed.
Run a live prepare after installing 0.18.0:

```bash
sqlx-js prepare
sqlx-js prepare --check
```

Projects using provider-aware schema workflows should generate and verify
against their configured source of truth:

```bash
sqlx-js dev --strict-inference
sqlx-js verify --strict-inference
```

If the project commits `.sqlx-js/schema/schema.json`, regenerate it separately
against the intended target database:

```bash
sqlx-js snapshot dump
sqlx-js snapshot check
```

Do not hand-edit `.sqlx-js/functions/functions.json` or
`sqlx-js-env.d.ts`. Earlier cache and snapshot formats fail closed with
regeneration guidance.

## Review function warnings

Warnings do not fail prepare or `--strict-inference`. Review every
`function-contract` diagnostic before deployment, especially:

- `SECURITY DEFINER` without a function-local trusted `search_path` ending in
  `pg_temp`;
- `SECURITY DEFINER` owned by a superuser;
- `SECURITY DEFINER` still executable by `PUBLIC`;
- application functions declared `LEAKPROOF`;
- `VOLATILE PARALLEL SAFE` declarations.

The metadata cannot prove that schemas preceding `pg_temp` are writable only by
trusted roles, that remaining `EXECUTE` grants are least-privilege, or that the
function body honors its volatility, leakproof, and parallel-safety labels.

## Owner and ACL parity

Owner attributes and ACLs are part of the generated contract. A shadow database
created under a different routine owner, or without the same `REVOKE` / `GRANT`
statements, intentionally fails `verify`.

Keep ownership and grants in schema source:

```sql
ALTER FUNCTION public.read_secret(uuid) OWNER TO app_function_owner;
REVOKE ALL ON FUNCTION public.read_secret(uuid) FROM PUBLIC;
GRANT EXECUTE ON FUNCTION public.read_secret(uuid) TO app_runtime;
```

Use the corresponding `ALTER PROCEDURE`, `REVOKE`, and `GRANT` forms for
procedures.

## Watch behavior

`prepare --watch` still reuses unchanged query fingerprints, but it refreshes
the function catalog on every filesystem-triggered prepare. This prevents
security metadata and warnings from remaining stale while the watch session is
running.

## Rollback

Version 0.17.x cannot consume the new committed function cache or schema
snapshot. To roll back the package:

1. install the previous sqlx-js version;
2. remove the generated `.sqlx-js/` directory and declaration output;
3. run that version's live `prepare`;
4. rerun that version's `snapshot dump` if the project commits a snapshot;
5. review and commit the downgraded artifacts atomically with the dependency
   rollback.

## Verification checklist

1. Run live `prepare` or provider-aware `dev`.
2. Review every `function-contract` warning.
3. Run `prepare --check --strict-inference`.
4. Run provider-aware `verify --strict-inference`.
5. Run `snapshot check` when snapshots are committed.
6. Run the application typecheck and test suite.
