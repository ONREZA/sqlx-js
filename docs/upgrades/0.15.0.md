# Upgrade to 0.15.0

Status: unreleased.

This guide applies to applications upgrading from `0.14.x` to `0.15.0`.
Because sqlx-js is still pre-1.0, this breaking runtime redesign is released as
a minor version under the repository's Rust-style pre-1.0 SemVer policy.

## Summary

| 0.14.x contract | 0.15.0 contract | Required action |
| --- | --- | --- |
| `getClient()` / `setClient()` | Removed | Own a scoped `createSqlClient()` or a separate raw `createClient()` |
| `SqlClient.client` | Removed | Move raw Postgres.js work to an explicitly owned raw client |
| One fixed Postgres.js pool | Managed pool generations | Do not retain or expose the managed pool |
| Query timeout covered only selected runtime work | End-to-end operation deadline | Configure `operationTimeoutMs` where a bounded request is required |
| Transaction timeout covered the callback after `BEGIN` | Deadline covers bootstrap through `COMMIT`/`ROLLBACK` | Handle `TransactionTimeoutError.outcome` |
| `SqlClient.ready()` / `close()` had no options | Bounded `ready()`, `ping()`, and `close()` | Update wrappers and test placeholders |
| No per-definition cancellation | `AbortSignal` and per-query deadlines | Pass execution options through `run()` or `runWith()` |

## Managed and raw clients are separate ownership boundaries

`createSqlClient()` now owns every Postgres.js pool generation it creates. A
timed-out generation can be poisoned, retired, and replaced without exposing a
stale pool reference to application code. The managed client therefore no
longer exposes `.client`, and the global `getClient()` / `setClient()` exports
are removed.

Before:

```ts
import { close, createClient, ready, setClient, sql } from "@onreza/sqlx-js";

setClient(createClient(process.env.DATABASE_URL, { max: 10 }));
await ready();
await sql(`SELECT 1`);
await close();
```

After:

```ts
import { createSqlClient } from "@onreza/sqlx-js";

const db = createSqlClient(process.env.DATABASE_URL, { max: 10 });
await db.ready({ timeoutMs: 5_000 });
await db.sql(`SELECT 1`);
await db.close({ graceMs: 5_000, forceAfterMs: 10_000 });
```

Use `createClient()` only when Postgres.js-specific methods are genuinely
required. It is a raw escape hatch: the caller owns `end()`, and sqlx-js does
not provide managed deadlines, generation recovery, lifecycle state, or
name-based `typeCodecs` for it.

```ts
import { createClient } from "@onreza/sqlx-js";

const raw = createClient(process.env.DATABASE_URL, { max: 1 });
try {
  await raw.unsafe("SELECT pg_advisory_unlock_all()");
} finally {
  await raw.end({ timeout: 0 });
}
```

Do not create a raw client merely to recover the removed `.client` property.
Managed and raw clients are intentionally independent pools with independent
lifecycle ownership.

## Three timeout layers have different scopes

```ts
const db = createSqlClient(process.env.DATABASE_URL, {
  connect_timeout: 5,
  statementTimeoutMs: 10_000,
  operationTimeoutMs: 15_000,
  cancelGraceMs: 1_000,
});
```

- `connect_timeout` bounds connection establishment and authentication.
- `statementTimeoutMs` installs PostgreSQL `statement_timeout` on each
  connection and bounds server execution after SQL reaches PostgreSQL.
- `operationTimeoutMs` bounds the complete managed path: runtime codec
  bootstrap, pool/connect wait, parameter encoding, execution, and result
  decoding.
- `cancelGraceMs` bounds the best-effort cancellation/rollback confirmation
  window before the generation is forcibly retired.

`operationTimeoutMs` is opt-in. Choose it at the application boundary; one
default cannot safely cover both latency-sensitive requests and long-running
jobs.

A managed timeout throws `QueryTimeoutError` with `timeoutMs`, `phase`,
`outcome`, `queryId`, and `generation`. A PostgreSQL `statement_timeout`
remains a database error with SQLSTATE `57014`.

`outcome: "not_sent"` means the application statement was not handed to the
driver. Once it has been handed off, the outcome is always `"unknown"` because
the statement may have completed. sqlx-js never retries either case
automatically. In particular, do not replay a timed-out `INSERT`, `UPDATE`, or
`DELETE` when the outcome is `"unknown"`.

## Per-query deadlines and AbortSignal

Named and mapped definitions accept execution options as the last `run()`
argument:

```ts
await findUser.run(db.sql, { id }, {
  timeoutMs: 2_000,
  signal: request.signal,
});
```

Positional definitions use `runWith()` because a trailing object can itself be
a PostgreSQL parameter:

```ts
await findUserById.runWith(
  { timeoutMs: 2_000, signal: request.signal },
  db.sql,
  id,
);
```

Execution options fail closed when the supplied executor cannot honor the
managed contract. They are not silently ignored by structural test doubles or
third-party adapters. Inside a transaction, a query-level timeout or abort
expires the entire scoped transaction so the connection can be rolled back
before reuse.

## Transaction deadlines now cover the complete transaction

`sql.transaction({ timeoutMs, signal }, callback)` starts its deadline before
runtime codec bootstrap. It covers pool acquisition, `BEGIN`, transaction
characteristics, the callback, every statement, `COMMIT`, and `ROLLBACK`.

```ts
await db.sql.transaction({
  isolation: "serializable",
  timeoutMs: 30_000,
  signal: request.signal,
}, async (tx) => {
  await tx.execute(`UPDATE jobs SET claimed = true WHERE id = $1`, jobId);
});
```

`TransactionTimeoutError.outcome` is:

- `"rolled_back"` only when Postgres.js confirms that the transaction returned
  to a clean state within `cancelGraceMs`;
- `"unknown"` when `BEGIN`, `COMMIT`, or `ROLLBACK` cannot be confirmed. The
  whole pool generation is retired in this case.

A transaction-scoped executor is valid only while its callback is active.
Using a captured `tx` after commit or rollback now fails locally without
dispatching SQL.

JavaScript cannot preempt arbitrary synchronous callback work or undo external
side effects. Long-running non-database work should observe its own signal and
remain idempotent.

## Lifecycle and observability additions

Managed clients add:

```ts
await db.ready({ timeoutMs: 5_000 });
await db.ping({ timeoutMs: 1_000 });

const state = db.snapshot();
// generation, state, activeOperations,
// lastSuccessAt, lastTimeoutAt, recycleCount

await db.close({ graceMs: 5_000, forceAfterMs: 10_000 });
```

`ready()` bounds codec discovery. `ping()` executes `SELECT 1` through the same
managed bootstrap, pool, deadline, and observer path as application queries.
`close()` is terminal for a scoped client: it stops admission immediately,
waits the grace period, interrupts remaining promises, and forces pool shutdown
within the total `forceAfterMs` bound. Repeated and reentrant close calls share
one shutdown.

New hooks are available on `createSqlClient()`:

- `onQueryStart(event)`;
- `onQueryTimeout(event)`;
- `onClientStateChange(event)`;
- `onLifecycleHookError(error, event)`.

Lifecycle events intentionally omit SQL text and parameters. They contain the
stable query ID/name, duration where applicable, generation, phase, and
outcome. Observer failures never replace a database result or database error.

The managed state machine exposes `healthy`, `poisoned`, `recycling`, `failed`,
`closing`, and `closed`. It does not claim unstable internal Postgres.js stages
such as queued, connected, or executing.

## Error handling additions

Applications may need to add branches for:

- `QueryTimeoutError` — the current operation exceeded its deadline;
- `QueryAbortedError` — its signal was aborted;
- `GenerationRecycledError` — a collateral operation belonged to a poisoned
  generation;
- `ClientClosingError` — shutdown rejected or interrupted the operation;
- `TransactionTimeoutError` — the complete transaction deadline expired.

Use `outcome`, not the error class alone, when deciding whether an application
operation can be attempted again. The library never retries SQL automatically.

## TypeScript migration

Every structural `SqlClient` placeholder must remove `client` and implement the
new methods:

```ts
const unconfigured: SqlClient<Registry> = {
  sql: unconfiguredSql,
  unsafe: () => missingDatabaseUrl(),
  ready: () => missingDatabaseUrl(),
  ping: () => missingDatabaseUrl(),
  snapshot: () => missingDatabaseUrl(),
  close: async () => {},
};
```

For a generated registry with `runtimeTypes`, raw
`createClient<Registry>()` requires explicit numeric Postgres.js `types`.
Managed `createSqlClient<Registry>()` continues to accept either name-based
`typeCodecs` or matching numeric `types`.

This release does not change the generated cache or declaration format. A
`sqlx-js prepare` regeneration is not required solely for the runtime upgrade,
although the normal live `prepare --verify` gate should still run before
deployment.

## Rollout checklist

1. Replace `getClient()`, `setClient()`, and `SqlClient.client` usage.
2. Update test doubles and unconfigured placeholders with `ping()` and
   `snapshot()`.
3. Add an application-specific `operationTimeoutMs`; keep
   `statementTimeoutMs` as the independent server-side limit.
4. Handle `unknown` outcomes without retrying non-idempotent SQL.
5. Pass HTTP or job cancellation through `AbortSignal` where useful.
6. Wire `ready()`/`ping()` into startup and readiness checks.
7. Call bounded `close()` from shutdown handling and await it.
8. Run the project typecheck, tests, `prepare --verify`, and a packaged runtime
   smoke against PostgreSQL before rollout.

Do not deploy application code that depends on the new methods while some
instances still resolve sqlx-js `0.14.x`. Upgrade the package and its client
wrapper atomically.
